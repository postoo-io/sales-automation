# 콜 요약 및 플러그 기록 (Call Summary)

> ⚠️ **사전 조건**: 이 스킬은 `sales-setup`이 완료돼 있어야 합니다 (최소 레벨 **L1**). 미달 시 0단계 점검에서 차단합니다.
> 발송·발행물에 회사명·발신자 서명이 포함되므로 활성 영업 담당자(`team[active]`)와 `company.displayName`이 반드시 채워져 있어야 합니다.

영업 콜·미팅 내용을 구조화하여 요약하고, 플러그 의뢰 히스토리에 메모를 남기고, 다음 액션 아이템을 플러그 Todo로 생성합니다. **쓰기 작업(히스토리 저장·Todo 생성)은 반드시 사용자 승인 후에만 실행합니다.**

---

## 1. 입력 형태 (3가지)

| 유형 | 설명 | 예시 |
|------|------|------|
| **녹음 파일** | mp3 / m4a / wav 로컬 파일 경로 | `/Users/me/recordings/call_acme_20260521.m4a` |
| **텍스트 트랜스크립트** | STT 변환이 완료된 텍스트를 붙여넣기 | Notta, CLOVA Note, Whisper 출력 결과 |
| **거친 노트** | 콜 중 작성한 메모·불릿 포인트 | "가격 우려, PoC 요청, 다음주 데모 제안" |

의뢰 ID가 없으면 먼저 묻습니다. 의뢰 ID가 있어야 플러그 맥락(회사명·담당자·단계)을 파악해 요약의 톤과 내용을 맞출 수 있습니다.

---

### 0단계 — 사전 점검 (setup gate)

스킬 진입 전 setup이 충분한지 확인합니다. 미달이면 진행하지 말고 사용자에게 `sales-setup` 실행을 안내합니다.

```bash
skills/sales-setup/scripts/check_setup.py --level L1 --quiet
# exit 0 = pass, exit 1 = 차단 (사용자에게 sales-setup 실행 안내)
```

이 스킬의 요구 레벨: **L1** (Pluuug 자격증명 + team[active] name·email·signature + company.displayName). 실패 시 사용자에게 보여줄 안내:

> "이 스킬을 실행하려면 sales-setup이 필요합니다. 부족한 항목이 있어 진행할 수 없습니다.
> 자세한 부족 항목은 `skills/sales-setup/scripts/check_setup.py --level L1` 출력을 참고하세요."

## 2. STT 처리 절차 (녹음 파일인 경우)

Claude Code는 오디오를 직접 STT 변환하지 않습니다. 아래 순서로 처리합니다.

### 단계 1 — 트랜스크립트 유무 확인

> "트랜스크립트(텍스트 변환본)가 이미 있나요? 있으면 바로 붙여주세요."

트랜스크립트가 있으면 STT 단계를 건너뛰고 바로 요약으로 이동합니다.

### 단계 2 — 로컬 Whisper 확인 (트랜스크립트가 없을 때)

```bash
which whisper
```

오디오 변환은 `scripts/transcribe.sh <파일경로>`로 자동화합니다 (whisper 자동 감지·로컬 실행, 미설치 시 대안 안내). 자세한 처리는 스크립트 주석 참조.

## 3. 의뢰 컨텍스트 수집

트랜스크립트(또는 노트)와 의뢰 ID가 확보되면 플러그에서 맥락을 가져옵니다.

```bash
# 의뢰 상태·회사·담당자·단계 조회
scripts/pluuug.py GET /v1/inquiry/<의뢰ID>

# 최근 히스토리 확인 (중복 저장 방지)
scripts/pluuug.py GET /v1/inquiry/<의뢰ID>/text_history
```

조회 결과에서 다음을 추출합니다:
- `companyName` — 고객사명 (요약 헤더에 사용)
- `inCharge` — 담당 영업 매니저
- `status` / `folder` — 현재 영업 단계
- `text_history` 최신 항목 — **1시간 내 동일 요약이 있으면 "중복일 수 있습니다" 경고**

## 4. 요약 추출 구조

트랜스크립트·노트를 분석해 아래 항목을 한국어로 추출합니다. **콜에서 언급되지 않은 내용은 임의로 추가하지 않습니다.**

출력 템플릿은 `assets/summary_template.md`(화면 표시용)와 `assets/pluuug_history_body.md`(의뢰 히스토리용) 두 가지를 사용. 사용자에게는 풀 템플릿을, 플러그에는 컴팩트 버전을 저장.

```markdown
## 콜 요약 — [회사명] / [날짜]

**콜 일시:** YYYY-MM-DD HH:MM  
**참석자:** [우리측] / [고객측]  
**통화 길이:** (파악된 경우)  

### 핵심 논의
- [논의 항목 1]
- [논의 항목 2]
- … (3–6개)

### 고객 요구사항 / 우선순위
- [우선순위 1]
- [우선순위 2]

### 반론 · 우려 · 이의제기
- [우려 사항] — [대응 내용 또는 미해결 상태]

### 합의 / 결정 사항
- [합의 항목]

### 액션 아이템
| 담당 | 내용 | 기한 |
|------|------|------|
| 우리 | [할 일] | YYYY-MM-DD |
| 고객 | [할 일] | YYYY-MM-DD |

### 다음 콜 · 미팅 약속
- [날짜/시간, 형태, 참석자]

### 영업 단계 변동 신호
- [예: "PoC 단계 이동 가능성 언급", "예산 승인 대기 중" 등]
```

## 5. 저장 절차 (사용자 승인 필수)

### 5-1. 사전 미리보기

요약과 Todo 후보를 먼저 사용자에게 보여줍니다. **이 시점까지 어떤 쓰기 호출도 하지 않습니다.**

```
─────────────────────────────────────────
[미리보기] 플러그에 저장될 내용
─────────────────────────────────────────

① 의뢰 히스토리 메모 (POST /v1/inquiry/{id}/text_history)
┌──────────────────────────────────────┐
│ ## 콜 요약 — 에이콘 / 2026-05-21    │
│ ...요약 전문...                       │
└──────────────────────────────────────┘

② Todo 후보 목록 (POST /v1/todo × N건)
┌────┬──────┬────────────────────────┬────────────┬──────┐
│ #  │ 유형 │ 액션                   │ 기한       │ 메모 │
├────┼──────┼────────────────────────┼────────────┼──────┤
│ 1  │ CA   │ 제안서 초안 공유       │ 2026-05-23 │ ...  │
│ 2  │ M    │ 데모 미팅 확정         │ 2026-05-28 │ ...  │
│ 3  │ MS   │ 계약서 초안 메시지 전달│ 2026-05-30 │ ...  │
└────┴──────┴────────────────────────┴────────────┴──────┘

이대로 저장할까요? (저장 / 수정 / 건너뛰기)
─────────────────────────────────────────
```

### 5-2. 의뢰 히스토리 저장

사용자가 **저장**을 선택하면 실행합니다.

```bash
scripts/pluuug.py POST /v1/inquiry/<의뢰ID>/text_history \
  --json '{"content":"## 콜 요약 — 에이콘 / 2026-05-21\n\n**콜 일시:** ...\n..."}'
```

- `content` 값은 마크다운 그대로 줄바꿈(`\n`)을 유지합니다.
- 플러그 화면에서 가독성 있게 표시되도록 섹션 구분에 빈 줄을 넣습니다.
- 1건만 POST합니다 (요약 전체를 하나의 히스토리 메모로).

### 5-3. Todo 생성

액션 아이템마다 개별 POST 합니다. 액션 아이템 → Todo 매핑 패턴은 `references/action_item_examples.md` 참조 (좋은 액션 vs 나쁜 액션 비교 포함).

```bash
# 예시 1 — 전화 후속
scripts/pluuug.py POST /v1/todo \
  --json '{"inquiry":<의뢰ID>,"type":"CA","dueDate":"2026-05-23","action":"제안서 초안 공유 후 피드백 요청","actionHistory":"콜에서 가격 검토 후 제안서 원한다고 함"}'

# 예시 2 — 미팅 약속
scripts/pluuug.py POST /v1/todo \
  --json '{"inquiry":<의뢰ID>,"type":"M","dueDate":"2026-05-28","action":"PoC 데모 미팅 진행","actionHistory":"다음주 수요일 오후 데모 진행 합의"}'

# 예시 3 — 메시지 전달
scripts/pluuug.py POST /v1/todo \
  --json '{"inquiry":<의뢰ID>,"type":"MS","dueDate":"2026-05-30","action":"계약서 초안 카카오톡 전달"}'
```

#### Todo 유형 선택 기준

| 유형 코드 | 의미 | 사용 상황 |
|-----------|------|-----------|
| `CA` | 전화 | 콜백, 전화 확인, 전화 후속 |
| `M` | 미팅 | 데모·발표·대면 미팅 약속 |
| `MS` | 메시지 | 카카오·슬랙·문자 전달 |
| `E` | 기타 | 위 3가지에 해당 없는 업무 |

#### Todo 필드 규칙

| 필드 | 필수 | 규칙 |
|------|------|------|
| `inquiry` | 필수 | 의뢰 ID (숫자) |
| `type` | 필수 | `CA` / `M` / `MS` / `E` |
| `dueDate` | 필수 | `YYYY-MM-DD`. "다음주 화요일"처럼 상대 표현이면 절대 날짜로 변환. 오늘 기준: 2026-05-21 |
| `action` | 필수 | 한 줄 핵심 (80자 이하) |
| `inCharge` | 선택 | 담당자 이름 (없으면 생략) |
| `dueTime` | 선택 | `HH:MM` (특정 시각이 정해진 경우) |
| `actionHistory` | 선택 | 콜에서 나온 맥락 (선택, 1–2문장) |

## 7. 가드레일

- **자동 저장 금지.** `POST /v1/inquiry/{id}/text_history` 와 `POST /v1/todo` 는 사용자가 **"저장"** 을 명확히 선택한 후에만 호출합니다.
- **추측 액션 아이템 금지.** 콜에서 실제로 논의되거나 합의된 항목만 Todo로 만듭니다. 스스로 "이건 해야 할 것 같다"는 이유로 Todo를 추가하지 않습니다.
- **외부 STT 자동 업로드 금지.** 녹음 파일을 클라우드 STT 서비스로 자동 전송하지 않습니다. 로컬 Whisper만 사용하거나, 사용자가 직접 변환한 텍스트를 받습니다.
- **중복 저장 경고.** `GET /v1/inquiry/{id}/text_history` 결과에서 최신 항목이 1시간 이내이면: "최근 1시간 내 히스토리가 이미 있습니다. 중복 저장일 수 있습니다. 계속 저장할까요?"
- **날짜 변환 명시.** "다음주 화요일" 같은 상대 표현을 절대 날짜로 변환할 때, 변환 결과를 사용자에게 보여주고 확인받습니다. (예: "다음주 화요일 → 2026-05-26, 맞나요?")
- **의뢰 ID 미확인 시 중단.** 의뢰 ID 없이는 플러그 저장 단계로 넘어가지 않습니다. 요약 자체는 생성할 수 있으나, 저장 전에 ID를 반드시 확보합니다.
- **키/시크릿 출력 금지.** API Key, Secret Key를 메시지·로그에 노출하지 않습니다.

## 8. 다른 스킬과의 연계

| 연계 스킬 | 관계 | 활용 방법 |
|-----------|------|-----------|
| **call-prep** | 선행 | 이 스킬 실행 전에 call-prep으로 콜을 준비. call-prep의 "다음 단계" 출력이 이 스킬의 입력 맥락이 됨 |
| **email-send** | 후행 | 콜 요약 완료 후 "팔로업 메일도 보낼까요?" 제안. 요약의 '합의 사항'과 '다음 약속'이 이메일 본문의 기초가 됨 |
| **daily-briefing** | 노출 | 이 스킬로 생성된 플러그 Todo가 다음날 daily-briefing의 Todo 리스트에 노출됨. `GET /v1/todo?inquiry_id=<id>&is_complete=false` 로 조회 |
| **pluuug-api** | 의존 | 의뢰 조회·히스토리 저장·Todo 생성 모두 pluuug-api의 호출 패턴을 따름. 인증·레이트 리밋·가드레일은 pluuug-api 스킬 규칙을 그대로 적용 |

### 팔로업 메일 연계 흐름 (예시)

```
콜 종료
  └─▶ call-summary 실행
        ├─ 요약 생성
        ├─ 플러그 히스토리 저장 (승인)
        ├─ Todo 생성 (승인)
        └─ "팔로업 메일 초안 만들까요?" 제안
              └─▶ email-send 실행 (사용자가 원할 때)
```
