# Account Research (고객사 리서치 & 리드 유효성 검증)

Pluuug 인입 데이터를 기반으로 고객사 정보를 수집하고, 부족한 정보를 파악한 뒤 리드가 유효한지 판단합니다.
허위/스팸 의심 신호도 함께 제공하되 최종 판단은 항상 사용자에게 맡깁니다.

---

## 입력으로 받을 수 있는 형태

| 입력 형태 | 예시 |
|-----------|------|
| Pluuug 의뢰 ID | "의뢰 1234 리서치해줘" |
| 회사명 + 담당자 이메일 | "acme 고객, contact@acme.co.kr 리서치" |
| 도메인만 | "acme.co.kr 이 회사 뭐하는 곳이야?" |
| 회사명만 | "(주)에이스테크 어떤 회사야?" |

세 가지를 조합해서 줄수록 정확도가 올라갑니다. 아무것도 없으면 먼저 묻습니다.

---

## 절차

### 1단계 — Pluuug 의뢰 데이터 수집

의뢰 ID가 주어지면 아래 순서로 순차 호출합니다.

```bash
# 의뢰 상세 (본문·fileSet·fileLinkSet·funnel·fieldSet·client 포함)
scripts/pluuug.py GET /v1/inquiry/{id}

# 의뢰 하위 히스토리 (4개 병렬 호출 가능)
scripts/pluuug.py GET /v1/inquiry/{id}/text_history
scripts/pluuug.py GET /v1/inquiry/{id}/email_history
scripts/pluuug.py GET /v1/inquiry/{id}/email_reply_history
scripts/pluuug.py GET /v1/inquiry/{id}/submit_history
```

의뢰 JSON 안에 `client.uniqueId`가 있으면 고객 상세도 가져옵니다.

```bash
# 고객 상세 (companyName, ceoName, businessRegistrationNumber, companyAddress,
#             contact, email, taxInvoiceEmail, content, businessType, businessClass)
scripts/pluuug.py GET /v1/client/{client_id}
```

의뢰 ID 없이 회사명만 있는 경우 고객 검색 먼저 시도합니다.

```bash
scripts/pluuug.py GET /v1/client?search=<회사명>
```

### 2단계 — 도메인·회사명 기반 웹 리서치

WebSearch와 WebFetch를 사용합니다. **사업자등록번호는 웹 검색 쿼리에 직접 넣지 않습니다**
(국세청 사업자 진위확인: https://www.nts.go.kr/nts/cm/cntnts/cntntsView.do?mi=6594&cntntsId=7946).

```
검색 쿼리 순서:
1. "<회사명> OR <도메인>" → 홈페이지, 회사 소개
2. "<회사명> 뉴스" → 최근 90일 이내 뉴스·투자·IR
3. "<회사명> 채용 OR 사람인 OR 잡코리아" → 채용 규모·성장 신호
4. "<담당자명> <회사명> LinkedIn OR 링크드인" → 담당자 직책·배경
5. "<도메인> site:nicebizinfo.com OR crefia.or.kr" → 기업신용 정보
```

WebFetch로 홈페이지 About·IR 페이지를 직접 파싱해 업종·사업 내용·고객층을 보강합니다.

### 3단계 — 갭 분석 & 리드 유효성 스코어 산출

수집한 데이터를 바탕으로 아래 점수표를 적용합니다.

#### 리드 유효성 점수표

| 신호 | 조건 | 점수 |
|------|------|------|
| 도메인 존재 | 도메인 확인·홈페이지 응답 | +20 |
| 회사 공개 정보 | NICE·사람인·잡코리아 등에서 회사 확인 | +15 |
| 사업자번호 기재 | client.businessRegistrationNumber 존재 | +10 |
| 요구사항 구체성 | 의뢰 본문에 수량·일정·예산·목적 중 2개 이상 | +15 |
| 담당자 실명 확인 | 이름+이메일 도메인 일치 (회사 도메인) | +10 |
| 회사 채용·뉴스 신호 | 최근 90일 내 뉴스·채용 공고 확인 | +10 |
| 이메일 수발신 기록 | email_history 또는 email_reply_history 존재 | +10 |
| 무료 이메일 도메인 | gmail·naver·daum·kakao 등 개인 메일 | -10 |
| 연락처 미상 | contact·email 모두 없음 | -10 |
| 의뢰 본문 1줄 이하 | submit_history 내용이 극도로 짧음 | -10 |
| 동일 IP·이메일 중복 | 짧은 기간 내 동일 정보로 다수 의뢰 | -15 |

**총점 → 등급**

| 점수 | 등급 | 의미 |
|------|------|------|
| 70–100 | A — 유효 | 즉시 영업 진행 권장 |
| 40–69 | B — 추가 확인 | 부족 정보 보완 후 판단 |
| 0–39 | C — 허위/스팸 의심 | 근거 명시 후 사용자 최종 결정 |

## 비즈니스 프로필 가중치 (선택)

`sales-setup`이 저장한 `profile.json`이 있으면 리드 적합도 평가에 자사 컨텍스트를 반영합니다.

- `keywords.goodFit` 키워드가 의뢰 본문·회사 정보에 매칭되면 **+5점**씩 (최대 +15)
- `keywords.outOfScope` 키워드가 매칭되면 **-10점**씩 — 우리 영역이 아닌 리드는 명시적으로 감점
- `brand.targetCustomers`와 회사 산업이 부합하면 리포트 끝에 "✅ 핵심 타깃 부합" 표시 (+5점 추가 가산)
- 모든 가중치는 어떤 키워드가 +/- 작용했는지 근거로 표시 (불투명한 점수 조정 금지)

비즈니스 프로필이 없으면 가중치 없이 베이스 룰만 적용합니다. 조회: `skills/sales-setup/scripts/profile.py --show`.

## 가드레일

- **의뢰 상태 변경·메모 자동 기록 금지**: 리포트 작성 후 "결과를 의뢰 히스토리에 메모로 남길까요?" 한 번만 묻습니다. 사용자가 승인하면 아래 호출을 실행합니다.
  ```bash
  scripts/pluuug.py POST /v1/inquiry/{id}/text_history --json '{"content":"[리서치 요약]"}'
  ```
- **허위문의 판단은 참고용**: "C 등급" 판단도 사용자가 최종 확인합니다. 자동 드롭·삭제는 하지 않습니다.
- **사업자등록번호 웹 노출 금지**: 번호를 검색 쿼리에 직접 넣지 않고, 국세청 공식 진위확인 URL을 안내합니다.
- **쓰기 작업은 pluuug-api 가드레일 준수**: POST/PATCH는 사용자 명시 승인 후에만 실행합니다.

> **표준 변수 참조**: 리포트 내 담당자 서명 등은 `{{me.name}}`, `{{me.signature}}`, `{{client.companyName}}` 같은 표준 변수로 채워집니다. 자세한 변수 명세는 `skills/sales-setup/references/template_variables.md` 참조.

## 사용하는 Pluuug 엔드포인트

| 엔드포인트 | 메서드 | 용도 |
|-----------|--------|------|
| `/v1/inquiry/{id}` | GET | 의뢰 상세 (본문·funnel·fieldSet·client) |
| `/v1/inquiry/{id}/text_history` | GET | 의뢰 텍스트 메모 이력 |
| `/v1/inquiry/{id}/email_history` | GET | 의뢰 이메일 발신 이력 |
| `/v1/inquiry/{id}/email_reply_history` | GET | 의뢰 이메일 수신 이력 |
| `/v1/inquiry/{id}/submit_history` | GET | 의뢰 폼 제출 이력 |
| `/v1/client/{id}` | GET | 고객 상세 (사업자번호·주소·업종 등) |
| `/v1/client?search=<회사명>` | GET | 회사명으로 고객 검색 |
| `/v1/inquiry/{id}/text_history` | POST | 리서치 결과 메모 저장 (사용자 승인 시) |

## 다른 스킬과의 연계

- **call-prep**: account-research 리포트를 그대로 call-prep에 전달해 콜 어젠다·질문지를 생성합니다. "이 리서치 결과로 콜 준비해줘"로 인계.
- **email-send**: 갭 분석의 부족 정보 항목을 바탕으로 고객에게 추가 정보 요청 메일 초안을 작성합니다. "부족 정보 요청 메일 써줘"로 인계.
- **pluuug-api**: 모든 Pluuug API 호출은 이 스킬의 `scripts/pluuug.py`를 거칩니다. 서명·인증 로직을 직접 구현하지 않습니다.
